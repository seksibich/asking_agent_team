# 输出格式规范

## 输出根目录

从记忆/偏好读取 `report_output_dir`，本地/通用环境默认工作文件根为 `盯盘/`。业务报告和普通临时产物写工作文件区；人格与记忆只能写记忆区，禁止混放。

## 本地/通用目录结构

```text
盯盘/
├── yyyy年MM月dd日/                       # 定时任务日报
│   ├── 01-盘前汇总.md
│   ├── 04-当日总结.md
│   └── 05-综合复盘.md
├── 投研/                                  # 用户分析、选股、调研及手动时段报告
├── 选股/                                  # 调度器正式选股报告
├── 周报/
├── 月报/
├── tmp/                                   # 普通临时脚本、文档、转换和中间产物
│   └── tmp_YYYYMMDD-HHmmss_文件名
└── agent记忆/
    ├── MEMORY.md                          # 永久规范、稳定偏好、已验证经验及索引
    ├── USER.md                            # 用户稳定资料与长期偏好
    ├── 服务状态与能力.md                  # 版本、文档同步、功能索引与接口约定
    ├── 关注与持仓.md                      # portfolio_version + 服务端全量镜像
    ├── 短期记忆/                          # 一事一文件及关联附件，解决或过期即删除
    ├── daily/                             # 每日业务快照
    ├── predictions.jsonl                  # 业务审计镜像
    └── 学习日志-yyyy年MM月.md             # 回测与配置审计
```

普通临时脚本、临时文档、格式转换和中间产物统一命名为 `tmp/tmp_YYYYMMDD-HHmmss_文件名`，不得写入 `agent记忆/`。只有为解决某一报错而需要跨轮次保存的接口报错、响应摘录等证据，才可放入 `短期记忆/`，且必须复用 `YYYYMMDD-HHmm-有效至YYYYMMDD-HHmm-描述` 事项前缀并登记关联；事项解决、证伪或到期时连同附件删除。

## Coze 双栏映射（强制）

- 左侧“工作文件”：日报、`投研/`、`选股/`、`周报/`、`月报/`、复盘和 `tmp/`；普通临时产物仍使用 `tmp/tmp_YYYYMMDD-HHmmss_文件名`。
- 右侧“记忆”：`基础设定/SOUL.md`；`所有对话/主对话/MEMORY.md`、`USER.md`、`关注与持仓.md`、`服务状态与能力.md`；`所有对话/主对话/recent_memory/` 等价于逻辑 `短期记忆/`。
- `SECRET.md` 或平台密钥区只由 Coze 安全机制管理；报告、普通记忆和短期记忆均不得保存真实密钥。
- 平台目录名不同也必须保持“工作文件/记忆”职责隔离，不得为了路径便利交叉存放。

## ★ 分析指令输出规则（重点）

当**用户主动向 Agent 发起分析指令**（如「帮我分析下光伏板块」「研究一下 XX 股」「挖一下最近的涨价方向」），产出**不进日报目录**，而是进独立目录：

- **行业/主题/事件研究**：`投研/yyyyMMdd-xx研究报告/`
- **方向选股**（用户指定事件、行业导向或热门板块）：`投研/yyyyMMdd-{主题}选股/`
- **单股调研**：`投研/yyyyMMdd-{股票名}个股调研/`
- `yyyyMMdd`：当天日期（如 20260714）
- `{主题}`/`{股票名}`：使用可辨识的简短中文名称，由 Agent 根据指令自动命名
- 目录内至少含 `研究报告.md`（主体）；数据量大时拆分 `数据附录.md`、`打分表.md`
- 同一天同一主题多次分析，目录后加序号：如 `20260714-光伏行业研究报告-2`
- 上述普通研究默认 `ephemeral`；用户正式选股只写服务端 manual 快照，不改 daily、predictions 或短期事项。用户要求持续跟踪时再上传为 watch。

**判定「分析指令」**：用户使用「分析/研究/深挖/看看/评估/挖掘」等主动请求，且针对具体标的、板块、主题或事件。定时任务自动产出的日报**不属于**分析指令，仍进日期目录。

## ★★ 触发来源决定输出目录（定时 vs 手动，重点）

**竞价和盘中盯盘没有定时来源，只允许用户显式触发；其他保留任务按来源隔离：**

| 触发来源 | 输出目录 | 是否写自动化记忆 |
|---|---|---|
| **保留定时任务**（`schedule.md` 的 T1/T2/T3/W1/M1/P1） | 日期目录 `yyyy年MM月dd日/`、`周报/`、`月报/` | ✅ 按对应任务生成 daily、auto 选股、预判或学习日志 |
| **用户手动触发**（包括盘前、竞价、盘中、盘后） | **用户指令目录** `投研/yyyyMMdd-手动{技能名}/` | ❌ 不覆盖定时日报，不改写 daily、predictions、短期事项或 `category=auto` |

- **核心原则**：禁止自动触发竞价和盯盘；用户手动触发的一切产出进入 `投研/`，对自动化状态只读。手动盯盘响应结束后不得自动创建下一轮任务。
- 手动触发若用户明确要求登记（如「把这几只加入我的持仓/关注」），才按用户指令调用自选搜索与上传；仍不写 category=auto，历史 watch/holding 快照仅在用户明确要求回测时补记。
- 判定：由调度器/定时器发起 = 定时；对话里用户主动发起 = 手动。拿不准时按「手动」处理（更安全，不污染自动化目录与回测）。

## 用户指令 → 输出/动作速查

| 用户说 | Agent 输出/动作 | 落到哪里 |
|---|---|---|
| 「我持有 XX」 | 先搜索并从结果确认股票；缺持仓成本或手数时追问，字段完整后上传为持仓 | 服务端自选；成功后刷新 `agent记忆/关注与持仓.md` |
| 「把 XX 持仓改为 X 元、Y 手」 | 搜索确认股票，按真实成本与整数手数上传最新持仓状态 | 服务端按代码覆盖；成功后刷新镜像 |
| 「我卖完/清仓了 XX」 | 搜索确认股票；按用户选择移除，或转为关注；不自行推算已实现盈亏 | 服务端自选；成功后刷新镜像 |
| 「关注一下 XX」 | 搜索并从结果选择，上传为关注；只有明确要求历史回测时才额外登记 watch 快照 | 服务端自选；成功后刷新镜像 |
| 「分析/研究行业、板块、主题或事件」 | 出行业/事件投研报告；默认 ephemeral | `投研/yyyyMMdd-XX研究报告/` |
| 「按 XX 事件/行业/热门板块选股」 | 先动态题材映射，再量化/趋势筛选并按“题材/事件 → 个股”分组；默认 ephemeral | `投研/yyyyMMdd-{主题}选股/` |
| 「分析/研究/评估 XX 股」 | 启用 `stock-research` 做单股调研；默认 ephemeral | `投研/yyyyMMdd-{股票名}个股调研/` |
| 「把上述结果加入观察/持续跟踪」 | 搜索确认后上传为关注；仅在另行明确要求时补做 1/3/7/30 日观察性回测，永不进入 auto 调参 | 服务端自选 + `agent记忆/关注与持仓.md` 镜像 |
| 「帮我跑一下盘前/竞价/盘中/盘后复盘」 | 出对应时段报告（手动版，团队/主按原技能） | `投研/yyyyMMdd-手动XX/`（不进日期目录、不写自动记忆） |

> 用户告知持仓/关注/清仓属于**当前状态更新指令**：先在内存中形成草稿并调用 `portfolio_upload`，只有上传成功后才用响应 rows 覆盖 `关注与持仓.md`；默认不生成报告，也不创建历史回测快照，用户另外要求分析或回测时才执行。

## 命名规范

- 日报：数字前缀排序，名称排序即时间顺序
- 投研目录：`yyyyMMdd-主题+类型`
- 全部 Markdown 格式，UTF-8

## 面向用户的通俗中文表达（所有报告硬约束）

1. **先说结论，再讲依据**：首屏、摘要、正文和推送使用普通投资者能直接理解的中文，句子简短，明确回答“发生了什么、影响谁、为什么关注、何时失效”。
2. **不得堆砌英文技术名称**：接口函数名、请求参数名、JSON 字段、数据库字段、内部类别和因子代码不得直接出现在首屏、结论、策略、风险或正文表格中。例如不得直接展示 `buy_weight_hint`、`category=watch`、`mom_12_1`、`trend_ma`、`z-score` 等。
3. **统一转成中文业务含义**：上述内部名称在用户报告中分别表达为“建议出手权重”“已加入观察”“中期动量”“均线趋势”“标准化相对强弱分”等；内部枚举 `ephemeral/watch/auto/holding` 分别写成“仅本次报告/观察跟踪/系统自动候选/用户持仓”。
4. **英文技术名只允许出现在三个位置**：数据来源附录的“来源接口”、接口失败与降级诊断、用户明确要求查看技术参数时。即使保留，也必须紧邻中文解释，不得连续罗列参数。
5. **正文只保留决策所需数字**：详细参数、完整接口返回和原始因子代码移入“数据附录”或省略；正文优先展示中文指标名称、当前值、相对位置、结论和证伪条件。
6. **表头和章节名必须中文化**：不使用英文参数名充当字段标题；“关键因子”应展示中文含义，原始代码仅可在附录建立一次“中文名称—内部代码”对照。
7. **用 emoji 做视觉锚点（推荐，适度）**：一眼结论的四类关键信息与主要章节标题前加统一 emoji，帮助用户一眼定位重心——📊 仓位/倾向、🔥 题材/事件、🎯 关注个股/候选、⚠️ 风险/证伪、📋 复盘总结、📰 消息面、💼 持仓、👀 关注、🌡️ 情绪、💰 资金、📈 趋势/涨价、🔗 数据来源、🛰️ 数据状态。emoji 只作高亮，不替代文字含义；同一类信息全局使用同一图标，避免堆砌或滥用。
8. **信息重心前置**：越能帮用户做决策的内容越靠前——复盘总结、关键消息面、选股/持仓/关注结论优先于筛选过程、数据口径与来源清单。用户下拉一次以内就能读到全部结论。

## 报告与推送分离（T1/T2/T3 硬约束）

1. T1 盘前、T2 17:30、T3 22:00 的 Markdown 报告必须在可核验数据范围内尽可能详尽，**不得为了推送字数删减报告正文**。
2. **所有 T1/T2/T3、趋势/量化选股、用户主动方向选股和单股调研报告，标题后第一节固定为 `## 一眼结论（核心摘要）`**，至少先展示：仓位/次日倾向；重点题材/事件 Top N；按“题材/事件 → 个股”分类的重点个股；最大风险/证伪。其后才是「目录导读」和详细证据；目录每一项必须对应实际章节。
3. 接口不可用时仍保留对应章节，明确写出缺失字段、失败接口、fallback、重试轨迹和实际数据日期；不得静默删除章节，也不得把“无数据”解释为“无风险/无事件”。
4. 推送是独立产物，只摘取重点，建议不超过 500 字；不得复制整篇报告，也不得只写“报告已生成”。推送必须给出任务/日期、仓位或次日倾向、1~3 条核心主线/事件、重点候选或持仓风险、完整报告路径、数据降级提示。

## 报告通用结构

> 📌 **信息重心铁律**：越有决策价值的信息越往前放。用户打开报告，第一屏就要看到「该做什么、盯什么、怕什么」，无需下拉、无需读技术细节。数据口径、来源清单、原始参数一律靠后。

1. 标题（含日期、类型、主题）
2. 🎯 **`## 一眼结论（核心摘要）`**（标题后第一节，最贵的信息放这里）：仓位/次日倾向、重点题材/事件 Top N、按「题材/事件 → 个股」分类的重点个股、最大风险/证伪。
3. 🧭 **目录导读**（逐项链接或列出后续实际章节）
4. 🛰️ 数据状态与口径（交易日、实际数据日期、接口状态、fallback、缺失项）
5. 📄 详细正文（按任务模板完整展开）
6. 🔗 数据来源与时间戳清单
7. ⚠️ 风险提示、证伪条件与未验证项

### 首屏任务语义（每类报告「一眼结论」先答什么）
- **T1 盘前** ☀️：今日关注，先给今日仓位倾向、隔夜/盘前重点事件、事件映射个股与风险。
- **T2 盘后** 🌆：当日最强题材及次日延续初判，先给次日倾向、最强题材阶段和代表股。
- **T3 复盘** 🌙：今日复盘 + 晚间新增事件 + 次日正式候选，先给正式次日方向与证伪。
- **趋势/量化/用户主动选股** 🎯：先按题材/事件分组展示结论和评分，再给筛选过程。
- **单股调研** 🔍：先给该股仓位/次日倾向、事件受益关系、最大风险，再给证据。

### 风险提示写作约定
- **PE/PB 归入风险提示**：仅作过往业绩对应的估值背景，**不作为看多、入选或上涨动能依据**。
- 涨价/价格类结论若仅单一来源，标「未交叉验证」并列入风险。
- 预期驱动标的明确预期证伪后的回撤风险；传闻标「传闻，待证实」。
- 业绩增长、预增、略增、扭亏、续盈等公告只代表可核验事实，**不得宣称必然利好**，不得单独替代四维与趋势判断。

## T1/T2/T3 完整 Markdown 模板

三类报告标题后第一节必须是 `## 一眼结论（核心摘要）`，并在本节内按固定顺序先写：仓位/次日倾向 → 重点题材/具体事件 Top N → “题材/事件 → 个股”重点个股 → 最大风险/证伪 → 首屏结论表。之后才能出现目录和正文。
