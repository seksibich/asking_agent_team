# 测试、探针、监控与运维

## 三类健康接口

| 接口 | 用途 | 依赖访问 | 失败语义 |
|---|---|---|---|
| `/live` | 判断进程是否存活 | 不访问数据库和外部数据源 | 进程不可达才失败 |
| `/ready` | 判断是否可以接收生产流量 | 数据库、Tushare 配置、功能数量、模块装载 | 任一关键项失败返回 503 |
| `/health` | Agent 和管理员诊断、版本协调 | 返回完整业务快照 | 为兼容始终返回诊断体，不作为流量探针 |

Docker healthcheck、远端部署验收和正式监控均使用 `/ready`。`/health` 继续承担市场阶段、日终状态、版本和量化盯盘摘要的兼容职责。

## 正式环境监控

`deploy/monitor_service.py` 提供两个动作：

- `probe`：每五分钟请求 `/live` 与 `/ready`，失败以非零状态退出，由 systemd/journald 记录。
- `daily`：每天 22:50 使用进程环境中的管理员 Key 调用 `/admin/monitor/daily`，不打印 Key。

配套单元：

- `stock-agent-monitor.service` / `.timer`
- `stock-agent-monitor-daily.service` / `.timer`

`remote_deploy.sh` 会渲染、安装并启用这些 timer。未获得用户明确 push 授权时，不会运行远端部署。

## 内存护栏与连接复用（v2.7.0 运维加固）

- **内存护栏**：主服务 systemd 单元由 `remote_deploy.sh` 自动写入 `MemoryHigh`（软上限，默认总内存 70%）、`MemoryMax`（硬上限，默认总内存 85%）与 `OOMPolicy=kill`。超过硬上限只在本服务 cgroup 内 OOM 并经 `Restart=on-failure` 自动拉起，避免小机型跑全市场 `precompute` 时 OOM 拖垮整机与 SSH。可用 `MEMORY_MAX`/`MEMORY_HIGH` 覆盖；总内存不可解析时退回 `infinity`（不设护栏，保持旧行为）。Docker 部署不走 systemd，此护栏仅对裸机/阿里云生效。
- **HTTP 连接复用**：`common.get_session()` 提供全局共享 `requests.Session`（keep-alive + 连接池 + 有限重试）。盘中实时快照的新浪分页并发抓取与通知 webhook 均复用连接，降低每轮 TCP/TLS 握手开销、稳定盯盘单轮延迟。池大小 `HTTP_POOL_MAXSIZE`（默认 16）、重试 `HTTP_RETRY_TOTAL`（默认 1）可调。
- **NaN 序列化修复**：`service/app.py` 的 `_versioned` 用 `_json_safe` 递归把 `NaN/Inf` 清洗为合法 JSON `null`，修复 `macro_*`/`fundamental_*`/`hot_ths`/`market_limit` 等接口的 HTTP 500。

## 预计算僵尸任务自愈（v2.7.0）

预计算在后台线程运行；服务进程被杀（部署重启、崩溃、OOM）时线程终止，但 DB 中的 `precompute_jobs` 行可能停留在 `running`，表现为「卡在某百分比」并占用全局单例、阻塞后续预计算。三层自愈：

1. **优雅停止标记（`precompute.abort_active_job`）**：`app.py` shutdown 事件把本进程仍在跑的任务标记为 `failed/任务中断`。覆盖部署重启（systemd SIGINT / docker SIGTERM）这一最常见路径，重启后不留僵尸 running。
2. **启动回收（`db.reap_stale_precompute_jobs(0)`）**：服务启动时回收所有无归属的活跃任务（新进程内不可能有在跑的预计算线程）。覆盖硬崩溃后重启的兜底。仅适用单实例；多 worker 需收敛到单进程执行。
3. **周期巡检（`daily_scheduler._loop` 每轮 `reap_stale_precompute_jobs(10)`）**：不受市场阶段限制，回收心跳超过 10 分钟的活跃任务。覆盖「进程存活但工作线程死亡」及 SIGKILL/OOM 未走优雅停止的情况，超时后自动自愈，无需手动触发或等待下一个 `final` 阶段。

失败为终态，不阻塞新预计算；`claim_precompute_job` 仍保留认领时的心跳超时回收作为并发兜底。

## 每日 Agent 可读汇总

原始低基数事件写入 `data/logs/monitor/events/YYYY/MM/DD.jsonl`，日报写入 `data/logs/monitor/daily/YYYYMMDD.md`。内容包括：

- 接口总量、业务拒绝、服务故障、状态分布、P50/P95/最大耗时；
- 慢、拒绝或故障最多的业务；
- 量化盯盘触发次数、状态分布、扫描覆盖、达标数和重点异动；聚合历史按交易日保留 30 个自然日，监控需关注连续交易日无消息、清理失败和历史查询异常；
- 日志压缩/删除结果、磁盘占用、容量告警和最近错误摘要。

监控按以下口径降噪：普通 `400`、`401`、`402`、`403`、`404`、`405`、`422` 等记为“业务拒绝”，单独出现不把整体状态改为“需关注”；`408`、`429` 和全部 `5xx` 记为“服务故障”。量化盯盘持续相同异常最多每五分钟记录一次，错误变化时立即记录。

## 日志生命周期与容量

每天生成汇总时，同时对 `data/logs/` 下的审计与监控日志执行生命周期维护：

| 环境变量 | 默认值 | 作用 |
|---|---:|---|
| `MONITOR_LOG_CLEANUP_ENABLED` | `true` | 是否执行压缩和删除；故障调查期间设为 `false` 可冻结证据 |
| `MONITOR_LOG_RETENTION_DAYS` | `90` | 达到该年龄的按日日志和日报删除 |
| `MONITOR_LOG_COMPRESS_AFTER_DAYS` | `7` | 达到该年龄的原始 JSONL 压缩为 gzip |
| `MONITOR_DISK_WARN_PERCENT` | `85` | 磁盘使用率告警线 |
| `MONITOR_DISK_MIN_FREE_GB` | `5` | 最低可用空间告警线 |

日期判断使用日志路径中的上海自然日，不依赖文件修改时间。压缩先写临时文件再原子替换；单个文件失败只记录错误，不影响日报和业务请求。容量任一阈值触发时，日报整体状态为“需关注”。当前策略只管理本机文件，不替代对象存储备份、云日志归档和删除审批。


## 覆盖边界

自动化测试覆盖市场时段、前后端静态契约、就绪判断、监控告警分级、日志压缩与保留、选股并发、动态 Key 摘要存储和筛选事务回滚。量化盯盘变更还需验证：当日查询、次日回看、周末回退、显式历史日期、未来/非法日期拒绝、30 日清理边界、历史页不被 WebSocket 覆盖，以及管理员默认盯盘/访客默认情绪。仍需在真实 MySQL/RDS、真实 Tushare 权限和浏览器中完成集成烟测；日志外部归档与恢复演练也需在正式基础设施中验证。
