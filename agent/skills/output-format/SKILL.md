# 输出格式规范

## 输出根目录

从记忆/偏好读取 `report_output_dir`，默认根目录为 `盯盘/`。所有文件写在该根目录下。

## 目录结构

```

├── yyyy年MM月dd日/                       # 定时任务日报
│   ├── 01-盘前汇总.md
│   ├── 02-竞价速报.md
│   ├── 03-早盘总结.md
│   ├── 04-当日总结.md
│   └── 05-综合复盘.md
├── 投研/                                  # ★ 用户指令产出（分析指令 + 手动触发时段技能，均归此）
│   ├── 20260714-XX研究报告/
│   │   ├── 研究报告.md
│   │   ├── 数据附录.md
│   │   └── 打分表.md

│   ├── 20260715-某某股份个股调研/          # 用户主动单股调研
│   └── 20260715-手动盘前汇总/              # 用户手动触发时段技能（不进日期目录）

│   ├── yyyyMMdd-趋势选股.md
```

## ★ 分析指令输出规则（重点）

当**用户主动向 Agent 发起分析指令**（如「帮我分析下光伏板块」「研究一下 XX 股」「挖一下最近的涨价方向」），产出**不进日报目录**，而是进独立目录：

- **行业/主题/事件研究**：`投研/yyyyMMdd-xx研究报告/`
- **方向选股**（用户指定事件、行业导向或热门板块）：`投研/yyyyMMdd-{主题}选股/`
- **单股调研**：`投研/yyyyMMdd-{股票名}个股调研/`
- `yyyyMMdd`：当天日期（如 20260714）
- `{主题}`/`{股票名}`：使用可辨识的简短中文名称，由 Agent 根据指令自动命名
- 目录内至少含 `研究报告.md`（主体）；数据量大时拆分 `数据附录.md`、`打分表.md`
- 同一天同一主题多次分析，目录后加序号：如 `20260714-光伏行业研究报告-2`
- 上述用户主动研究默认均为 `ephemeral`：只落报告，不写 predictions/daily/观察池、不调用 `log_selection`；仅用户明确要求持久化时转为 `watch`

**判定「分析指令」**：用户使用「分析/研究/深挖/看看/评估/挖掘」等主动请求，且针对具体标的、板块、主题或事件。定时任务自动产出的日报**不属于**分析指令，仍进日期目录。

## ★★ 触发来源决定输出目录（定时 vs 手动，重点）

**同一个时段技能（盘前/竞价/盘中/盘后），输出目录由「谁触发」决定，绝不混放：**

| 触发来源 | 输出目录 | 是否写自动化记忆 |
|---|---|---|
| **定时任务**（`schedule.md` 注册的 T1~T7 / W1 / M1 / D1 / P1，由调度器触发） | 日期目录 `yyyy年MM月dd日/`、`周报/`、`月报/` | ✅ 生成/更新 `daily/yyyyMMdd.md` 快照、`log_selection(category=auto)` 登记、写 predictions/学习日志 |
| **用户手动触发**（对话中要求「帮我跑一下盘前/竞价/复盘/盯盘一下」） | **用户指令目录** `投研/yyyyMMdd-手动{技能名}/`（如 `投研/20260715-手动盘前汇总/`） | ❌ **不写自动化记忆**：不覆盖当日定时日报、不改写 `daily/yyyyMMdd.md`、**不以 category=auto 登记 log_selection**（避免污染回测调参） |

- **核心原则**：自动化链路的目录与记忆（日期目录、daily 快照、auto 回测样本）只由定时任务写；用户手动触发的一切产出，一律归到用户指令目录 `投研/`，且对自动化状态**只读**。
- 手动触发若用户明确要求登记（如「把这几只加入我的持仓/关注」），才按用户指令走记忆更新（category=watch/holding），仍不写 category=auto。
- 判定：由调度器/定时器发起 = 定时；对话里用户主动发起 = 手动。拿不准时按「手动」处理（更安全，不污染自动化目录与回测）。

## 用户指令 → 输出/动作速查

| 用户说 | Agent 输出/动作 | 落到哪里 |
|---|---|---|
| 「我持有 XX，买入 X 万」 | 更新持仓 + 买卖流水（累计买入总金额）；`log_selection(category=holding)` | `agent记忆/关注与持仓.md`（不出报告，除非用户要分析） |
| 「我卖了 XX，卖出 X 万」 | 累计卖出总金额，记流水；若清零→已清仓归档 + 已实现盈亏 | `agent记忆/关注与持仓.md` |
| 「我清仓了 XX」 | 移入「已清仓归档」，算已实现盈亏，停止持仓盯（除非要求转关注） | `agent记忆/关注与持仓.md` |
| 「关注一下 XX」 | 加入关注区；`log_selection(category=watch)`，重点盯直到取消 | `agent记忆/关注与持仓.md` |
| 「分析/研究行业、板块、主题或事件」 | 出行业/事件投研报告；默认 ephemeral | `投研/yyyyMMdd-XX研究报告/` |
| 「按 XX 事件/行业/热门板块选股」 | 先动态题材映射，再量化/趋势筛选并按“题材/事件 → 个股”分组；默认 ephemeral | `投研/yyyyMMdd-{主题}选股/` |
| 「分析/研究/评估 XX 股」 | 启用 `stock-research` 做单股调研；默认 ephemeral | `投研/yyyyMMdd-{股票名}个股调研/` |
| 「把上述结果加入观察/持续跟踪/纳入后续回测」 | 将明确指定标的转为 `watch`，补齐题材事件字段并做 1/3/7/30 日观察性回测；永不进入 auto 调参 | `agent记忆/关注与持仓.md` + `log_selection(category=watch)` |
| 「帮我跑一下盘前/竞价/盘中/盘后复盘」 | 出对应时段报告（手动版，团队/主按原技能） | `投研/yyyyMMdd-手动XX/`（不进日期目录、不写自动记忆） |

> 用户告知持仓/关注/清仓属于**记忆更新指令**，默认只更新 `关注与持仓.md`（并按需 `log_selection` watch/holding），不生成报告；用户另外要求分析时才出投研报告。

## 命名规范

- 日报：数字前缀排序，名称排序即时间顺序
- 投研目录：`yyyyMMdd-主题+类型`
- 全部 Markdown 格式，UTF-8

## 面向用户的通俗中文表达（所有报告硬约束）

1. **先说结论，再讲依据**：首屏、摘要、正文和推送使用普通投资者能直接理解的中文，句子简短，明确回答“发生了什么、影响谁、为什么关注、何时失效”。
2. **不得堆砌英文技术名称**：接口函数名、请求参数名、JSON 字段、数据库字段、内部类别和因子代码不得直接出现在首屏、结论、策略、风险或正文表格中。例如不得直接展示 `buy_weight_hint`、`category=watch`、`mom_12_1`、`trend_ma`、`z-score` 等。
3. **统一转成中文业务含义**：上述内部名称在用户报告中分别表达为“建议出手权重”“已加入观察”“中期动量”“均线趋势”“标准化相对强弱分”等；内部枚举 `ephemeral/watch/auto/holding` 分别写成“仅本次报告/观察跟踪/系统自动候选/用户持仓”。
4. **英文技术名只允许出现在三个位置**：数据来源附录的“来源接口”、接口失败与降级诊断、用户明确要求查看技术参数时。即使保留，也必须紧邻中文解释，不得连续罗列参数。
5. **正文只保留决策所需数字**：详细参数、完整接口返回和原始因子代码移入“数据附录”或省略；正文优先展示中文指标名称、当前值、相对位置、结论和证伪条件。
6. **表头和章节名必须中文化**：不使用英文参数名充当字段标题；“关键因子”应展示中文含义，原始代码仅可在附录建立一次“中文名称—内部代码”对照。

## 报告与推送分离（T1/T6/T7 硬约束）

1. T1 盘前、T6 17:30、T7 22:00 的 Markdown 报告必须在可核验数据范围内尽可能详尽，**不得为了推送字数删减报告正文**。
2. **所有 T1/T6/T7、趋势/量化选股、用户主动方向选股和单股调研报告，标题后第一节固定为 `## 一眼结论（核心摘要）`**，至少先展示：仓位/次日倾向；重点题材/事件 Top N；按“题材/事件 → 个股”分类的重点个股；最大风险/证伪。其后才是「目录导读」和详细证据；目录每一项必须对应实际章节。
3. 接口不可用时仍保留对应章节，明确写出缺失字段、失败接口、fallback、重试轨迹和实际数据日期；不得静默删除章节，也不得把“无数据”解释为“无风险/无事件”。
4. 推送是独立产物，只摘取重点，建议不超过 500 字；不得复制整篇报告，也不得只写“报告已生成”。推送必须给出任务/日期、仓位或次日倾向、1~3 条核心主线/事件、重点候选或持仓风险、完整报告路径、数据降级提示。

## 报告通用结构

1. 标题（含日期、类型、主题）
2. **`## 一眼结论（核心摘要）`**（标题后第一节；仓位/次日倾向、重点题材/事件 Top N、按“题材/事件 → 个股”分类的重点个股、最大风险/证伪）
3. **目录导读**（逐项链接或列出后续实际章节）
4. 数据状态与口径（交易日、实际数据日期、接口状态、fallback、缺失项）
5. 详细正文（按任务模板完整展开）
6. 数据来源与时间戳清单
7. 风险提示、证伪条件与未验证项

### 首屏任务语义
- **T1**：今日关注，先给今日仓位倾向、隔夜/盘前重点事件、事件映射个股与风险。
- **T6**：当日最强题材及次日延续初判，先给次日倾向、最强题材阶段和代表股。
- **T7**：今日复盘 + 晚间新增事件 + 次日正式候选，先给正式次日方向与证伪。
- **趋势/量化/用户主动选股**：先按题材/事件分组展示结论和评分，再给筛选过程。
- **单股调研**：先给该股仓位/次日倾向、事件受益关系、最大风险，再给证据。

### 风险提示写作约定
- **PE/PB 归入风险提示**：仅作过往业绩对应的估值背景，**不作为看多、入选或上涨动能依据**。
- 涨价/价格类结论若仅单一来源，标「未交叉验证」并列入风险。
- 预期驱动标的明确预期证伪后的回撤风险；传闻标「传闻，待证实」。
- 业绩增长、预增、略增、扭亏、续盈等公告只代表可核验事实，**不得宣称必然利好**，不得单独替代四维与趋势判断。

## T1/T6/T7 完整 Markdown 模板

三类报告标题后第一节必须是 `## 一眼结论（核心摘要）`，并在本节内按固定顺序先写：仓位/次日倾向 → 重点题材/具体事件 Top N → “题材/事件 → 个股”重点个股 → 最大风险/证伪 → 首屏结论表。之后才能出现目录和正文。

## 独立推送摘要模板（T1/T6/T7）

