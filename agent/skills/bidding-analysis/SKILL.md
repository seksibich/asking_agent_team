# 竞价分析（仅用户显式触发）

> 本 Skill 不属于任何定时任务。只有用户当前明确要求查看竞价时，主 Agent 才执行一次；不得在固定时点自动运行、主动推送、后台续跑或自行安排后续盯盘。
>
> 产出进入 `投研/yyyyMMdd-手动竞价分析/`，不覆盖自动日报，不改 predictions、daily、短期事项或 `category=auto`。完整加载本 Skill 不构成调用授权。

## 前置步骤

1. 调 `GET /health`，确认交易日、数据可用性和实际数据时间；竞价尚未结束时不得把未完成数据当最终结果。
2. 读取用户本次指定股票；只有请求明确涉及持仓、关注或盘前候选时，才读取服务端自选镜像和当日观察对象记忆。
3. 调 `market_timing`（市场择时）与 `sentiment_temperature`（情绪温度）；失败时如实披露，不得自行估算。

## 分析对象

合并去重后传给 `bidding_analysis` 的 `codes` 仅来自本次请求明确覆盖的对象：

- 用户指定股票。
- 用户明确要求纳入的关注、持仓、前一日正式候选或盘前高热度临时观察列表。
- 服务端另外返回全市场异常高开、竞价爆量和竞价成交额 Top20。

不得因为系统中存在关注、持仓或候选记录而自动触发竞价分析。

## 服务调用与返回口径

```json
{"function":"bidding_analysis","params":{"codes":["600XXX.SH","000XXX.SZ"],"top_n":20}}
```

服务返回四组结果：

- `targets`：本次关注标的的竞价表现。
- `market_top`：按竞价成交额排序的全市场 Top20。
- `abnormal_gap_top`：异常高开标的。
- `burst_volume_top`：竞价爆量标的。

重点字段包括：`gap_pct`（高开幅度）、`auction_amount`（竞价成交额）、`amt_ratio`（竞价额/昨日成交额）、`pre_close`（昨收）、`prev_amount`（昨日成交额）。字段缺失必须标注，不得推算替代。

## 分析流程

1. **量价匹配**：结合昨收和昨日成交额判断高开是否有量能支持；高开且竞价明显放量，表示开盘资金认可度较高；高开无量则保持谨慎。
2. **情绪与催化**：
   - 情绪回暖或冰点反转、存在经核验的正向催化、竞价同步放量，可列为“超预期候选”，但必须说明概率和失效条件。
   - 高位高开叠加连续高热，重点提示冲高回落风险，不追逐确定性结论。
3. **抄底概率**：只有连续冰点、标的已明显回撤、竞价企稳或温和放量，并有涨价或产业逻辑支撑时，才可提示“具备抄底观察条件”；不得直接给买入指令。
4. **开盘倾向**：结合 `buy_weight_hint`（建议出手权重）归纳为积极、观望或防守，并写明证据与证伪条件。
5. 如需与盘前判断比较，只记录本次可核验偏差；不得追加自动预判或写自动记忆。
6. 当前响应结束即任务结束，不自动调用 `intraday-watch` 或创建后续提醒。

## 严谨与风险要求

- 竞价数据必须来自 `bidding_analysis`；接口失败时标记数据缺失与 `degraded`（降级状态），不得用盘口猜测替代。
- “超预期”“抄底”均为概率性判断，必须同时给出情绪、事件、量价依据和失效条件。
- 竞价只代表开盘阶段意愿，必须提示“竞价强不等于全天强”。
- 涨价、业绩等事实遵守至少两个独立来源交叉验证；单一来源标“未交叉验证”。
- 不给确定性买卖指令，不因本次结果自动加入关注、持仓或创建短期事项。

## v2.2.0 当前调度与日终边界

- 现行 Agent 定时任务仅为 T1/T3/W1/M1/P1；竞价分析不在定时任务中，只能由用户当前明确请求后单轮执行。
- 服务端在交易日 16:00 自动完成日终收口；Agent 只读 `health.daily_finalize` / `precompute_status`，不得自动调用 `precompute_daily_factors`。
- 管理员预计算仅可在用户当前明确要求诊断或补数时单次手动调用，不得与竞价失败、自动续跑、Hook、cron 或 Agent 循环联动。
