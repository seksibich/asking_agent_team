# 修改完成后的运行与发布流程

本规则默认始终生效，适用于本工作区全部代码与文档修改。

## 本地完成动作

1. 修改与必要验证完成后，**不得自行创建 Git commit，不得自行 push**。
2. 根据改动范围判断是否需要重建或重启本地 Docker 服务：
   - 后端、前端、依赖、Docker 配置或运行时装载文件发生变化时，重建并重启受影响服务。
   - 仅修改不影响运行时的文档时，可以不重启。
3. 先检查本地 Docker 服务状态；若服务未运行，则主动启动服务。
4. 启动或重启后检查容器状态与 `/live`、`/ready`、`/health`，确认进程存活、生产流量就绪且诊断快照可用；失败时查看日志并修复。

## 提交、推送与阿里云部署

1. 只有用户明确要求 commit 或 push 时，才执行对应 Git 操作；“完成修改”“继续”等表述不视为授权。
2. 只有用户明确要求 push 时才推送远端，禁止自行 push。
3. push 成功后，**仅当改动涉及服务运行时（后端、前端、依赖、Docker 配置或运行时装载文件）时**，才通过 `/deploy` 流程同步部署到阿里云并检查远端健康状态。
4. **纯 agent 文档 / 文档改动不需要重新部署上云**：仅修改 `agent/**`（提示词、Skill、角色、记忆规范）、`doc/**`、`profile/CHANGELOG-AGENT.md` 等不影响服务运行时的文档时，push 完成即可，不触发阿里云部署；`agent_doc_version` 由 agent 自行经仓库/GitHub 对齐。
5. 未获得 push 授权时，仅保留本地修改和本地 Docker 验证结果，不触发阿里云部署。
